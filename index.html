FireLive

The idea for this project was to make a software for live mixing and live producing with integrated loop buffers, entirely customizable and usable with any custom controllers.

It's not complete as it lacks lots of features I have in mind, but the basics are there.

I made the first working version in 1 week and used it in a little 30 minute set with a friend, it was quite fun. I made it with Godot Engine to iterate fast on the UI side, and to use the Godot Audio Engine.

After the first week I refactored all my code, fix some bugs, polished some features, and start working on a C++ gdextension for all the computing intensive task, like generating the audio spectrum or adding additional audio effect.

Before Firelive I had already developped a little application game called Firestage.


It's a VJing app, not as advanced, where you can mess with light, smokes, fire and laser in front of a crowd, it's not really advanced, as it was supposed to be customizable, but it's fun to play.

I love live shows, lightning and audio techniques that hyped up a crowd, and I hope I can one day work on something like that.

Server Meshing

I have recently tried to create a scalable multiplayer-game networking architecture based on the idea of "not much replications, lots of users" (don't search on the internet I made that up). I tought that if I could achieve a cool mutliplayer MMO concept without complex replication or persistance,
I could have a never before seen number of players connected to a single world.

Quite a naïve idea, but projects must start somewhere.
I started with a simple game server and a proxy server to tell players which game server to connect to. Then I modified game servers to mesh into each other, the proxy server also helped the game servers to connects to each other. It was time to make some improvement, I added, basic security for client - server communication (pickling/serializing packets), player load balance and game servers redundancy. Everything worked great so I decided to add VOIP, at first I used an all purpose compression algorithms for the packets transmission which was terrible. To do this more efficiently Il learned that I had to use Opus, the industry open standards for voice/sound transmission, but Opus is not natively supported by godot. So I forked a C++ godot gdextension github repo that was almsot doing what I needed, and modified/optimized it to get what I wanted in a practical workflow. I was done with the architecture so I got an OVH server, and after tackling numerous bugs it worked !

I'm very proud of this project not because it led anywhere but because it was a real challenge that taught me a lot.

Elumin

Elumin is the biggest project I have worked on.

It's an atmospheric game in a fictional place, I worked on it alone on my free time for the past 4 years, writing and re-writing the story, making and re-making models, programming and re-programming features.
Over the years, and more recently I had to develop my own tools to fix performance issue, mainly 3D issue : draw calls count, lights and shadows, video memory. All of this led to terrible performance.
Some of the issues was deeply related to my workflow, which I wanted to preserve because of the ease of use and fast iterating capabilities.
To keep this workflow, I had to develop a blender add-on in python using the Blender API.
This add-on allowed me to assign countless amount of different materials to any parts of a model without having to bake textures.

It was an fascinating problem-solving game to extract every bit of performant I could. At the time, the project is not remotely near what I have in mind for it to be good, but it's the only project where I allow myself to not think about a deadline.

Elumin has a STEAM PAGE, maybe it will be complete one day.

Metronim

For this project I was inspired by all ".io" partygames sites, it was back in 2021 and I wanted to be the next skribl.io, I guess.

I was not very experienced in web development but it didn't appear too hard because I was already developing games and application for years.
It was a slow learning process and not my best work, especially on the graphical side, because I was doing everything in vanillaJS using a good old canvas.

At first it was not about train and explosion, but about hacking (more in the vein of a s0urce.io), but with a couple of friend.

There was a secret password that you wanted to guess from a player, another secret password that you wanted a player to guess, all of that with a "private" chat functionnality, and you had tools to see or hijack communication.

I remember that it was too hard to make the game fun and interesting, that's why I embraced ✨the train approach✨.
